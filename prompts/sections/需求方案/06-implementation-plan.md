# 06 实施计划 / section prompt

## anchor

本 section 是需求方案的收束段，给出项目实施的总体节奏（不是详细 WBS）。措辞偏"规划方向"，不偏"工序甘特"。依赖 `project_period` + `budget_range` 给出阶段划分与投入节奏。

## intake 字段依赖

- `project_period`（必填）
- `budget_range`（选填）

## CoT 生成思路

1. 引导句：本节给出项目总体实施节奏
2. 按 `project_period` 字面拆阶段（典型 3-4 阶段，按项目周期长短调整）：
   - 规划设计阶段
   - 建设开发阶段
   - 测试验收阶段
   - （若有运维期）运维保障阶段
3. 每阶段一段：起止时间（基于 `project_period` 推算 / 措辞"约 N 个月"不必精确日期）+ 主要工作内容 + 关键交付物
4. 若 `budget_range` 含建设期 / 运维期分项，最后可加一段投入节奏说明
5. 不写"X 月 X 日"具体日期，统一用"第 N 个月"或"建设期内"

## 输出格式

二级标题 + 引导句 + N 个三级标题阶段段 + 可选投入节奏段：

```markdown
## 实施计划

<引导句，约 50 字>

### 阶段一：规划设计

<段落 / 含起止时间 + 主要工作 + 关键交付物>

### 阶段二：建设开发

<段落>

### 阶段三：测试验收

<段落>

### 阶段四：运维保障（若有运维期）

<段落>

### 投入节奏（若 budget_range 含分项）

<段落>
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 不写具体日期，统一用"第 N 个月" / "建设期内"措辞
- 不展开 WBS / 不列具体任务清单（那是实施方案层级）
- 阶段数依据 `project_period` 长度合理选取，不刻意凑齐 4 阶段
