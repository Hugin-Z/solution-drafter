# 02 客户痛点 / section prompt

## anchor

本 section 是需求方案的"问题诊断"段，必须紧贴客户输入的痛点字面，不二次提炼 / 不分类归并。痛点是后续"功能需求" / "技术架构"的来源，源头失真整篇失真。

## intake 字段依赖

- `core_pain_points`（必填 list）
- `existing_systems`（选填 list / 用于关联痛点指向的具体系统）

## 生成思路

1. 引导句一句：点明本节是"对贵单位现有业务运行状况的诊断梳理"
2. 把 `core_pain_points` list 逐条渲染为 `-` 列表，每条字面保留用户原句结构，可补一句"该现状导致 ..."但不引入新事实
3. 若 `existing_systems` 出现，可在收束段提示"上述痛点与现有 <系统名> 的能力边界相关"，但不臆测哪个系统对应哪个痛点
4. 不做"分类"（如归到"数据" / "管理" / "技术"三类），保持用户原次序

## 输出格式

二级标题 + 引导句 + 列表 + 可选收束段：

```markdown
## 客户痛点

<引导句，约 50 字>

- <痛点 1 原句 + 现状影响补语>
- <痛点 2 原句 + 现状影响补语>
- ...

<可选收束段：与现有系统的关联，约 80 字>
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 痛点条目数 = `core_pain_points` 长度，不增不减
- 不"分类"不"归并"，保持原次序
- 字段缺失 → 列表位写"- 【待补充】"
