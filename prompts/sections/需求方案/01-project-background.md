# 01 项目背景 / section prompt

## anchor

本 section 是"需求方案"文档的开篇，定位客户所在的政策环境 / 行业阶段 / 项目立项动因。读者是客户中高层（信息化牵头领导 / 业务部门负责人 / 财务 / 评审专家），需要在 2 分钟内理解"这个项目为什么要做"。

## intake 字段依赖

- `project_name`（必填）
- `client_name`（必填）
- `client_industry`（必填）

## CoT 生成思路

1. 先点名客户单位 + 项目名（开篇句）
2. 客户所在行业的总体政策 / 业务背景（不引入用户未提及的具体政策文件名 / 不臆造）
3. 项目立项动因：用 `client_industry` 字段反向推导客户类型（区县级政府 / 央国企 / 等），但措辞限于"贵单位作为 <client_industry>"句式，不扩展
4. 收束于"本次建设的 <project_name> 即承接这一背景而提出"

## 输出格式

二级标题 + 2 段正文：

```markdown
## 项目背景

<段落 1：客户单位 + 项目名定位，约 100 字>

<段落 2：行业 / 政策背景 + 立项动因，约 200 字>
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 不引入用户未提及的政策文件名 / 编号 / 发文单位
- 不夸张表述（"国家战略" / "重大意义"等模板化词慎用，除非用户字面已用）
- 字段缺失 → 整段写"【待补充】"，不编造
