# 03 功能需求 / section prompt

## anchor

本 section 把 §02 客户痛点翻译成"功能需求条目"。每条痛点应映射到 1-2 条功能需求，措辞从"问题描述"转为"系统应具备的能力"。这里是需求方案承上启下的关键节点，措辞偏差会传导到后续技术架构。

## intake 字段依赖

- `core_pain_points`（必填 list）
- `existing_systems`（选填 list / 用于"对接 / 集成 <系统>"类需求）

## 生成思路

1. 引导句：本节将上节痛点转化为对新系统的功能要求
2. 对每条 `core_pain_points`，生成 1-2 条功能需求条目，措辞模板："具备 ... 能力" / "支持 ..." / "提供 ..."
3. 功能需求条目用三级标题 `### F1` / `### F2` / ... 编号
4. 每条功能需求下含 2-3 句正文：能力描述 + 与痛点的对应关系（"对应 §02 第 N 条痛点"可省略字面，但措辞要呼应）
5. 若 `existing_systems` 出现且与某痛点相关，可加一条"对接 / 集成"类功能需求

## 输出格式

二级标题 + 引导句 + N 个三级标题段：

```markdown
## 功能需求

<引导句，约 50 字>

### F1 <功能名>

<能力描述 1-2 句>

### F2 <功能名>

<能力描述 1-2 句>

...
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 功能需求条目数 ≥ `core_pain_points` 长度（每痛点 1-2 条 / 不少于痛点数）
- 不引入用户未提及的能力（如"AI 大模型推理" / "区块链存证"等，除非字面已在 痛点 / 现有系统中出现）
- 措辞偏需求侧（"应具备" / "支持"），不偏方案侧（"采用 X 框架实现"）
