# 05 非功能需求 / section prompt

## anchor

本 section 描述系统的非功能性指标（性能 / 可用性 / 安全 / 兼容性 / 扩展性等）。指标的具体数值依赖 §04 技术架构选型 + `project_period` + `budget_range` 推算的合理量级。M2 阶段不强制具体数值（"目标响应时间 < 2s"等可省略），保持"指标维度 + 定性目标"即可。

## intake 字段依赖

- `project_period`（必填 / 影响"可用性指标"的实施期 vs 运维期划分）
- `budget_range`（选填 / 间接影响"性能指标"上限）

## CoT 生成思路

1. 引导句：本节给出系统的非功能性要求
2. 按维度列出（不强制全列 / 按本项目特征挑相关的）：
   - **性能**：响应时间 / 并发用户数 / 数据吞吐量（量级方向，不必精确）
   - **可用性**：服务可用率 / 故障恢复时间 / 实施期 vs 运维期的分阶段目标（引用 `project_period`）
   - **安全**：等保级别 / 数据加密 / 权限管控（不引入用户未提的具体合规要求）
   - **兼容性**：浏览器 / 操作系统 / 移动端 / 与 `existing_systems` 的接口标准
   - **扩展性**：用户规模增长 / 数据量增长 / 功能模块增添的承载能力
3. 措辞模板："系统应满足 ..." / "应支持 ..." / "目标可用率不低于 ..."
4. 量级表述用"行业典型量级"措辞，不给具体数（除非客户字面已给）

## 输出格式

二级标题 + 引导句 + N 个三级标题维度段：

```markdown
## 非功能需求

<引导句，约 50 字>

### 性能

<段落 / 含量级方向>

### 可用性

<段落 / 引用 project_period>

### 安全

<段落>

### 兼容性

<段落 / 引用 existing_systems 接口标准>

### 扩展性

<段落>
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 不引入用户未提及的具体合规要求（如"等保三级" / "GDPR" / 具体国标编号）
- 不给精确数值（"500ms 响应" / "99.99% 可用率"）除非客户字面已声明
- 措辞偏需求侧（"应满足"），不偏方案侧（"采用 X 实现"）
