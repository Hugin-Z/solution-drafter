# 04 技术架构概述 / section prompt

## anchor

本 section 把功能需求翻译为"系统将由哪些主要技术组件构成"，是需求方案中唯一允许出现具体技术名词的段。仍偏需求侧（"采用 ... 技术路线"），不偏实施侧（不写部署拓扑 / 不写组件版本号）。非功能需求（§05）会引用本节架构选型作为评估基准。

## intake 字段依赖

- `existing_systems`（选填 list / 决定"对接现有系统"的范围）
- `reference_cases`（选填 list / 决定"参考某案例同类架构"的措辞）

## 生成思路

1. 引导句：本节给出系统的总体技术架构思路
2. 分层描述（典型四层 / 但不强制）：
   - 数据层：数据汇聚 / 存储选型方向
   - 平台层：基础平台 / 中台能力
   - 应用层：业务功能 / 用户交互
   - 集成层：与 `existing_systems` 的对接关系
3. 若 `reference_cases` 出现，可加一句"参考 <案例名> 的同类架构思路"，不展开案例细节
4. 不指定具体厂商 / 产品名 / 版本号

## 输出格式

二级标题 + 引导句 + 分层描述 + 可选图占位符：

```markdown
## 技术架构概述

<引导句，约 50 字>

### 数据层

<段落>

### 平台层

<段落>

### 应用层

<段落>

### 集成层

<段落 / 含与 existing_systems 的对接说明>

**【图 4.1: 总体技术架构示意图】**
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 不写具体厂商名 / 产品名 / 版本号（除非字面已在 `existing_systems` / `reference_cases` 出现）
- 不写部署拓扑 / 服务器规格（那是实施方案层级）
- 集成层措辞限于"与 <existing_systems 字面> 对接"
