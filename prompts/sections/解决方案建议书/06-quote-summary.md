# 06 报价概要 / section prompt

## anchor

本 section 是建议书的收束段，给出我方对项目报价的总体说明。区别于详细报价表（在附件 / 单独提交），本段只描述报价口径 / 阶段划分 / 主要项构成。

## intake 字段依赖

- `budget_range`（选填 / 客户预算口径 / 本方案报价应在此范围内或与之协调）
- `project_period`（必填 / 决定报价的"建设期 vs 运维期"划分）

## CoT 生成思路

1. 引导句：本节给出本次报价口径与构成
2. 报价总体口径段：基于 `budget_range` 给出本方案报价的总体框架（"本次报价在客户预算范围 <budget_range> 内"）
3. 阶段划分段：按 `project_period` 拆"建设期报价 vs 运维期报价"
4. 主要项构成段：典型 4-6 项（软件许可 / 硬件设备 / 系统集成 / 实施服务 / 培训 / 运维 / 等）/ 不给具体金额 / 只给项构成
5. 报价说明段：报价有效期 / 报价口径（含税 / 不含税）/ 付款节奏

## 输出格式

二级标题 + 引导句 + 4 个三级标题段：

```markdown
## 报价概要

<引导句，约 50 字>

### 报价总体口径

<段落 / 引用 budget_range>

### 阶段划分

<段落 / 引用 project_period>

### 主要项构成

- 软件许可
- 硬件设备（如有）
- 系统集成与实施
- 项目培训
- 运维服务（运维期）
- 其他（如有）

### 报价说明

<段落 / 报价有效期 / 含税口径 / 付款节奏说明>
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 不给具体金额（除非 `budget_range` 字面已含具体数 / 且作为客户预算引用 / 不作为最终报价）
- 主要项构成按本项目实际删减 / 不刻意凑齐 6 项
- 不写"低价中标" / "成本控制最优"等竞争口径
