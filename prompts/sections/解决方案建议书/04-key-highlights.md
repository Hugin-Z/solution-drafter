# 04 方案亮点 / section prompt

## anchor

本 section 突出我方方案相对竞品的差异化优势。是售前文档的"得分点"段，读者评审专家会用本段判断是否给高分。

**重要：方案亮点不是 stub 实现细节，是真实方案撰写逻辑——基于客户痛点 + 现有系统局限 + 行业参考案例反推方案差异化。M5 真实 LLM 接入时，本 prompt 引导 LLM 做该推理；M3 stub 阶段，demo-local stub 用模板字面拼装，不代表真实差异化分析。**

## intake 字段依赖

- `core_pain_points`（选填 list / 痛点是亮点的来源 / 每痛点对应 1 亮点）
- `existing_systems`（选填 list / 现有系统局限是亮点的反衬背景）
- `reference_cases`（选填 list / 行业案例是亮点的参照系）

## CoT 生成思路（M5 真实 LLM 接入时执行）

1. 从 `core_pain_points` 提取每条痛点的"现有方案的不足"
2. 从 `existing_systems` 提取"现有系统未覆盖 / 集成度低 / 三维场景未赋能"等局限
3. 从 `reference_cases` 提取"同类项目的常规做法"作为对照基线
4. 反推本方案的差异化亮点（典型 4-6 条 / 每条对应 1 痛点或 1 局限）
5. 措辞模板："针对 ..., 本方案 ..., 区别于 ..."

## 输出格式

二级标题 + 引导句 + N 个三级标题亮点段：

```markdown
## 方案亮点

<引导句，约 50 字>

### 亮点 1 <亮点标题>

<段落 / 含针对哪个痛点 / 区别于现有什么做法>

### 亮点 2 <亮点标题>

<段落>

...
```

## 素材运用（S2 / assets.json）

本 section 写作时必须读取 `assets.json` 的 `sections.<本 section id>.acquired`（S2 产出 / 见 `prompts/stages/s2-acquire.md`）：

- 每条 `status: "acquired"` 的素材，把其 `content` 实质性织入对应论点（支撑论述，而非堆砌）。
- 每条 `status: "待补充"`（S2 未获取到）的素材，在其应支撑的位置写「【待补充】」，不编造（红线 3）。
- 不在正文暴露 `source` / `asset_type` 等字段名（内部术语不外泄 / 红线 4）。

## 红线复述

- 亮点数 ≥ `core_pain_points` 长度（每痛点至少 1 亮点）
- 不引入用户未提及的"国产化" / "AI 大模型" / "区块链"等热门词（除非字面已在 intake 中出现）
- 措辞偏对比侧（"区别于" / "突破"），不偏自夸侧（"行业领先" / "首创"慎用）
